# AAX Build and Validation

AAX support is available on macOS and Windows when `aax` is listed in
`package.metadata.wrac.supported_formats`. Default build, install, and validate
commands use that metadata list; `--target=aax` requests only AAX.
Explicit AAX targets fail when `aax` is not listed in `supported_formats`.

## Prerequisites

- AAX SDK extracted locally.
- AAX Validator/DSH archive or extracted Validator/DSH directory.
- CMake and a platform C++ toolchain supported by clap-wrapper.

Set `AAX_SDK_ROOT` to the extracted AAX SDK root directory:

```sh
export AAX_SDK_ROOT=/path/to/aax-sdk-2-9-0
```

For validation, set one of these:

```sh
export AAX_VALIDATOR_DSH_ROOT=/path/to/aax-validator-dsh
# or
export AAX_VALIDATOR_DSH_ARCHIVE=/path/to/aax-validator-dsh-2024-6-0-138bab0d-mac-arm64.tar.gz
```

When neither Validator/DSH variable is set, `xtask` also checks the standard
download file names under `~/Downloads`.

## Metadata

AAX identity is generated from `[package.metadata.wrac]` in the plugin package
manifest. Before shipping, set these fields to product-owned values:

- `aax_manufacturer_id`
- `aax_product_id`
- `aax_categories`
- `aax_stem_configs`
- `supported_formats = ["clap", "vst3", "au", "aax"]`

Each AAX stem config needs a unique `plugin_id`. Keep manufacturer, product, and
stem plugin IDs stable after release because hosts use them for plugin identity
and project recall.

## Build

```sh
cargo xtask build --target=aax
# or build every format declared in supported_formats:
cargo xtask build
```

Use `cargo xtask build --target=aax --dry-run` to inspect the task graph and
dependency order without requiring the SDK or producing artifacts.

## Install

AAX plugins install to the system-wide Avid plugin folder on macOS and Windows.
With `--scope=default`, `xtask` installs AAX system-wide and CLAP/VST3/AU
user-locally.

```sh
cargo xtask install --target=aax
```

`--scope=user` is invalid for AAX. Use `--scope=system` when you want the
system-wide choice to be explicit.

`cargo xtask uninstall` defaults to `--scope=all`; for AAX this resolves to the
system-wide Avid plugin folder because there is no user-local AAX install scope.

## Validate

```sh
cargo xtask validate --target=aax
# or validate every format declared in supported_formats:
cargo xtask validate
```

The validator runs the selected AAX Validator tests by test ID and saves official
JSON results under `target/wrac-plugins/<package>/wrac/validation/aax/`.
`xtask` uses Avid's bundled DTT runner on both macOS and Windows because DTT is
the package's documented automation layer for DigiShell and behaves consistently
in local shells and hosted CI.
`xtask` fails when any selected test reports a non-pass `result_status`.
With `--continue-on-error`, failed AAX validation still returns a non-zero exit
status but allows independent package/format tasks to continue; downstream AAX
tasks that depend on the failed task are skipped.

The local validation target intentionally skips:

- `test.cycle_counts`: DSP/HDX cycle-count validation is outside this native
  local build target.
- `test.page_table.load`: page-table XML is not generated by this template.

Signing, notarization, Pace wrapping, installer creation, and product
distribution are outside this template's build scope.
